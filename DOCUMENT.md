## 1. ゲーム概要

**タイトル:** 見やすいWebページを作ろう

このゲームは、**Webデザインの基本原則をインタラクティブに学ぶ教育型ゲーム**です。
プレイヤーが色・フォント・余白・レイアウトを自由に操作して架空のWebページをデザインし、AIがそのデザインを多角的に評価します。

評価結果とAIのアドバイスを通じて、「なぜ読みやすいデザインが良いのか」をゲーム感覚で体験できます。

---

## 2. ゲームの目的と教育的意義

### 学習目標

| 学習項目 | 内容 |
|---|---|
| **視認性** | 色の対比（コントラスト）がテキストの読みやすさに与える影響 |
| **レイアウト** | 余白・文字揃え・情報の階層構造がUXに与える影響 |
| **認知負荷** | デザインの複雑さが読者の理解しやすさに与える影響 |

### なぜゲーム形式か

「自分でパラメータを動かして → リアルタイムで変化を確認 → AIに採点してもらう」というサイクルで、読みやすさの原則を**体験として**習得できます。

## 3. ゲームの流れ

```
タイトル画面（page-title）
    ↓ スタート！ボタン → デザインをランダム初期化
遊び方説明画面（page-how-to）
    ↓ ゲームを始めるボタン
ゲーム画面（page-game）  デザイン編集 + リアルタイムプレビュー
    ↓ AI診断を開始！ボタン
評価画面（page-evaluation）  AI採点 + レーダーチャート表示
    ↓ 用語・技術の解説ボタン
解説画面（page-explanation）  デザイン理論・AI技術の説明
```

### 各画面の詳細

#### タイトル画面（`page-title`）
- ゲームタイトル・サブタイトルを表示
- **モデルバッジ**：前回の接続状態（接続OK / 制限中 / エラー / テスト中）を表示
- **キーバッジ**：現在のAPIキー設定状態を表示
- スタートボタンでデザインをランダム初期化してゲーム開始

#### 遊び方説明画面（`page-how-to`）
- ゲームルールと評価基準（視認性・構成・認知負荷）の概要

#### ゲーム画面（`page-game`）
- 左パネル：デザイン調整コントロール（14種類のパラメータ）
- 右パネル：リアルタイムプレビュー
- ゲーム開始時にデザインはランダム初期化される

#### 評価画面（`page-evaluation`）
- ゲーム画面のプレビューをミニ表示（同期コピー）
- 3カテゴリのAI評価カード（視認性・構成・認知負荷）
- レーダーチャートによるスコア可視化（Chart.js）
- 総合スコア表示（3カテゴリの平均）

#### 解説画面（`page-explanation`）
- デザイン原則の解説（認知負荷・ゲシュタルト原則など）
- AI技術の解説（Transformer・言語モデルなど）
- プロンプトエンジニアリングの解説
- デザイン改善ヒント集

プレイヤーが操作できるデザイン要素は以下の14種類です。

### カラー設定（6項目）

10色のソフトカラーパレットから選択します。

| パラメータ名 | 対象 |
|---|---|
| `bgColor` | 背景色 |
| `textColor` | 本文テキスト色 |
| `titleColor` | タイトルテキスト色 |
| `cardColor` | カード背景色 |
| `cardTitleColor` | カードタイトル色 |
| `cardTextColor` | カード本文色 |

カラーパレット（`config.js` の `SOFT_COLORS`）：

| パラメータ名 | 対象 | 選択肢 |
|---|---|---|
| `layout` | テキスト揃え | left / center / right |

## 5. AI評価システム

### 評価カテゴリ

AIは3つの観点からデザインを**並列**かつ**独立**して評価します。

| カテゴリID | 評価内容 | プロンプト |
|---|---|---|
| `visibility` | 背景とテキストのコントラスト・WCAGアクセシビリティ | `prompt_visibility.txt` |
| `layout` | 情報の階層構造・余白の活用・テキスト揃え | `prompt_layout.txt` |
| `cognitive` | デザインの統一感・読者の精神的負担 | `prompt_cognitive.txt` |

### グレード判定基準

スコアはAIが0〜100で返し、グレードはサーバー側（`grade.py`）で判定します。

```python
def get_grade(score):
    if score >= 90: return "S"   # 卓越
    elif score >= 80: return "A" # 優秀
    elif score >= 70: return "B" # 良好
    elif score >= 60: return "C" # 普通
    else: return "D"             # 要改善
```

> グレードをAIに判定させないのは、出力のばらつきを防ぐためです。

### 評価フロー

```
フロントエンド（evaluator.js）
    ↓ POST /api/evaluate  { category, params, keyIndex/customKey, modelOrder }
    ↓ ※3カテゴリを Promise.all で並列送信（400ms ずつずらして開始）

バックエンド（main.py）
    ↓ prompt_body.txt + prompt_{category}.txt を結合
    ↓ HEXカラー → 日本語名に変換（白・紺・灰…）してプロンプトに埋め込み
    ↓ asyncio.to_thread でブロッキングSDK呼び出しを非同期化

AIモデル（優先順に試行）
    ↓ 1. gemma-4-26b-a4b-it
    ↓ 2. gemma-4-31b-it
    ↓ 3. gemini-3.1-flash-lite
    ↓ すべて失敗かつ500/429 → 5秒待機してリトライ（1回）
    ↓ JSON形式で評価結果を返答

バックエンド（main.py）
    ↓ Markdown記法を除去 → JSONパース
    ↓ grade.py でグレード判定
    ↓ { grade, score, reasons, advice, model, rate_limited } を返却

フロントエンド
    ↓ 評価カードにグレード・理由・アドバイスを表示
    ↓ レーダーチャートをリアルタイム更新
    ↓ タイトル画面のモデルバッジを更新（localStorageに保存）
```

### AIレスポンス形式

AIには以下のJSON形式で返答するよう指示します。

```json
{
    "score": 82,
    "reasons": ["背景と文字のコントラストが十分", "余白が適切に確保されている"],
    "advice": "カードの背景色をさらに明るくすると視認性が向上します。"
}
```

### JSONパース堅牢化

AIが返す文章にMarkdown記法（\`\`\`json\`\`\`）や余分なテキストが混じる場合があるため、正規表現でJSON部分のみを抽出します。

```python
def clean_json_string(text):
    cleaned = text.strip()
    match = re.search(r'\{.*\}', cleaned, re.DOTALL)
    if match:
        cleaned = match.group(0)
    return cleaned
```

## 6. フロントエンド設計

### アーキテクチャ

フロントエンドは **SPA（Single Page Application）** として動作します。
`index.html` は最小限のHTMLのみを持ち、`app.js` が起動時に6つのHTMLページ断片を動的にロードして結合します。

```
index.html（シェル）
    ↓ DOMContentLoaded
app.js
    ↓ fetch: pages/title.html, how-to.html, game.html,
    ↓        evaluation.html, explanation.html, settings.html
    → #page-container に結合して挿入
    → イベントリスナ登録・初期化
```

### 状態管理

アプリの実行時状態は `state` オブジェクトに集約します。

```javascript
const state = {
    designParams: { ...DEFAULTS }, // 現在のデザイン設定（14パラメータ）
    radarChart: null,              // Chart.js インスタンス
    scores: { visibility: 0, layout: 0, cognitive: 0 }
};
```

永続状態は localStorage で管理します（詳細は [8. APIキー管理](#8-apiキー管理) 参照）。

### ページ遷移

CSSクラスの付け外しで画面を切り替えます（実際のページ遷移なし）。

```javascript
function navigateTo(id) {
    document.querySelectorAll('.page').forEach(p => p.classList.remove('active'));
    document.getElementById(id).classList.add('active');
}
```

### モジュール構成

ES Modules（`import/export`）を使用し、責務を分離しています。

| ファイル | 責務 |
|---|---|
| `config.js` | デザインパラメータの定数・マッピング・パスワードハッシュのみ |
| `evaluator.js` | コントラスト計算・レーダーチャート・API通信・テストモック |
| `app.js` | ページロード・状態管理・UI制御・ナビゲーション・キー管理・モデル順序管理 |

### リアルタイムプレビュー

スライダーやカラーボタンの操作のたびに `updatePreview()` が呼ばれ、
`applyDesignToElement()` がプレビューDOMに直接CSSを適用します。

### テストモード

`localStorage.testMode === 'on'` のとき、`evaluator.js` はAPIを呼ばずにモックデータを返します。
全画面の動作確認をAPIキーなしで行えます。

```javascript
if (localStorage.getItem('testMode') === 'on') {
    return { grade: 'TEST', score: 0, reasons: [...], advice: '...' };
}
```

## 7. バックエンド設計

### エンドポイント一覧

| メソッド | パス | 内容 |
|---|---|---|
| GET | `/` | `index.html` を返す |
| GET | `/api/keys` | 設定済み環境変数キーの番号一覧（値は返さない） |
| POST | `/api/test` | 指定キーでモデルへの接続テスト |
| POST | `/api/evaluate` | デザイン評価リクエスト |
| GET | `/static/*` | 静的ファイル配信 |

### `/api/evaluate` の処理詳細

**リクエスト**

```json
{
    "category": "visibility",
    "params": { "bgColor": "#ffffff", "textColor": "#1e293b", ... },
    "keyIndex": 1,
    "modelOrder": ["gemma-4-26b-a4b-it", "gemma-4-31b-it", "gemini-3.1-flash-lite"]
}
```

**処理フロー**

1. `keyIndex` / `customKey` からAPIクライアントを生成（`resolve_api_key()`）
2. `modelOrder` が送られてきた場合はそちらを優先、なければデフォルトの `MODELS` を使用
3. プロンプトを組み立て（`prompt_body.txt` + `prompt_{category}.txt` + パラメータ埋め込み）
4. `asyncio.to_thread` で非同期化した上でモデルを呼び出し
5. 失敗時は次のモデルへ。全モデル失敗かつ500/429なら5秒待機してリトライ（1回）
6. 成功したらJSONパース → グレード判定 → レスポンス返却

### `/api/test` の処理詳細

指定キーで全モデルを試行し、最初に成功したモデル名とメッセージを返します。
失敗時はエラーコード・エラー種別・日本語原因・モデル別エラー一覧を返します。

```json
{
    "ok": false,
    "error_code": "403",
    "error_kind": "PERMISSION_DENIED",
    "error_cause": "アクセス権なし（Gemma利用規約未同意 または 組織ポリシーによる制限）",
    "model_errors": [
        { "model": "gemma-4-26b-a4b-it", "code": "403", "kind": "PERMISSION_DENIED" }
    ]
}
```

### フォールバック・リトライ設計

```python
for attempt in range(2):           # 最大2回試行
    for model_name in models:       # 優先順にモデルを試行
        try:
            response = await asyncio.to_thread(
                client.models.generate_content, ...
            )
            return 成功レスポンス
        except Exception as e:
            500/429 → transient_error = True
            continue
    if transient_error and attempt == 0:
        await asyncio.sleep(5)      # 5秒待機してリトライ
    else:
        break
return エラーレスポンス
```

## 8. APIキー管理

### 仕組み

APIキーはサーバーの環境変数で管理し、フロントエンドには値を渡しません。
フロントエンドは「どのキーを使うか」の指定のみを送信します。

```
フロントエンド: { keyIndex: 5 } または { customKey: "AIza..." }
    ↓
バックエンド: os.getenv("GEMINI_API_KEY_5") でキーを取得
```

### localStorageの保存内容

| キー | 内容 |
|---|---|
| `keyMode` | `"env"`（環境変数）/ `"custom"`（直接入力） |
| `keyIndex` | 選択中の環境変数キー番号（1〜20） |
| `customKey` | 直接入力したAPIキー |
| `modelOrder` | モデル優先順位（JSON配列）。未設定時はサーバーの `MODELS` を使用 |
| `testMode` | `"on"` でAI呼び出しをスキップ |
| `lastModelName` | 前回成功したモデル名（タイトル画面バッジに表示） |

### 環境変数の設定

```env
# .env（ローカル）または Vercel 環境変数
GEMINI_API_KEY=default_key       # キー未選択時のフォールバック
GEMINI_API_KEY_1=key_for_user_1
GEMINI_API_KEY_2=key_for_user_2
...
GEMINI_API_KEY_20=key_for_user_20
```

## 9. 設定画面

タイトル画面右上の歯車アイコンからアクセスします（パスワード保護あり）。
パスワードハッシュは `config.js` の `SETTINGS_PASSWORD_HASH`（SHA-256）で管理します。

### 機能一覧

#### 接続情報
- 使用モデルの優先順位テーブル（↑/↓ボタンで並び替え可能）
- API エンドポイント表示
- 接続ステータス表示

#### APIキー管理
- **方式1（環境変数キー）**：サーバーに設定された `GEMINI_API_KEY_{n}` をボタンで選択
- **方式2（直接入力）**：APIキーを貼り付けて保存（localStorageに保存）

#### テストモード
- ON にするとAIへのリクエストを送らずモックデータで全画面を確認できる
- 全画面に紫色のバナーが表示される

#### デバッグ操作
- **API疎通テスト**：現在のキー・モデルへの接続確認（エラー詳細も表示）
- **localStorage クリア**：保存された設定をすべてリセット
